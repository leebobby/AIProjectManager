## 1. 架构概述

```
                     ┌─────────────────────────────┐
  浏览器 60人  ──────▶│         Nginx               │
                     │  • 静态文件 (Vue 前端)        │
                     │  • 反向代理 /api → 后端       │
                     └────────────┬────────────────┘
                                  │
                     ┌────────────▼────────────────┐
                     │   Uvicorn + FastAPI 后端     │
                     │   监听 127.0.0.1:8000        │
                     └────────────┬────────────────┘
                                  │
                     ┌────────────▼────────────────┐
                     │   SQLite  (app.db)           │
                     │   config.json               │
                     └─────────────────────────────┘
```

**技术栈**
- 后端：Python 3.11 + FastAPI + Uvicorn
- 前端：Vue 3 + Vite（构建为静态文件，由 Nginx 直接服务）
- 数据库：SQLite（文件型，零维护）
- 反向代理：Nginx

## 3. 首次部署

> 以下所有命令在 **服务器** 上以 `ubuntu`（或其他普通用户）执行，需要 root 权限的步骤会显式使用 `sudo`。

### 3.1 安装系统依赖

```bash
sudo apt update && sudo apt upgrade -y

# Python 3.11
sudo apt install -y python3.11 python3.11-venv python3-pip

# Node.js 18 LTS（通过 NodeSource）
curl -fsSL https://deb.nodesource.com/setup_18.x | sudo -E bash -
sudo apt install -y nodejs

# Nginx & 工具
sudo apt install -y nginx git sqlite3
```

验证安装：

```bash
python3.11 --version   # Python 3.11.x
node -v                 # v18.x.x
nginx -v                # nginx/1.x.x
```

### 3.2 拉取代码

```bash
# 选择一个合适的目录，例如 /opt
sudo mkdir -p /opt/aipm
sudo chown $USER:$USER /opt/aipm

cd /opt/aipm
git clone <你的仓库地址> app
# 或者直接上传压缩包后解压：
# unzip aipm.zip -d app
```

目录结构应如下：

```
/opt/aipm/app/
├── backend/          ← Python 后端
│   ├── main.py
│   ├── requirements.txt
│   ├── config.json   ← 会在首次启动后自动创建或手动编辑
│   └── app.db        ← 数据库（首次启动后自动创建）
└── frontend/         ← Vue 前端源码
    ├── src/
    └── package.json
```

### 3.3 部署后端

```bash
cd /opt/aipm/app/backend

# 创建虚拟环境
python3.11 -m venv venv
source venv/bin/activate

# 安装依赖
pip install -r requirements.txt

# 编辑配置文件（关键：修改 issue_report_path 为服务器上的报表目录）
cp config.json.example config.json  # 如有示例文件
# 或者直接编辑：
nano config.json
```

`config.json` 示例：

```json
{
  "current_stages": ["BFI", "岳麓山", "明场"],
  "issue_report_path": "/opt/aipm/reports",
  "about_content": "岳麓山项目管理系统",
  "hero_badges": ["项目名称", "团队名称", "版本", "阶段"]
}
```

**创建 systemd 服务**（让后端开机自启并自动重启）：

```bash
sudo nano /etc/systemd/system/aipm-backend.service
```

填入以下内容：

```ini
[Unit]
Description=岳麓山项目管理系统后端
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/opt/aipm/app/backend
Environment="PATH=/opt/aipm/app/backend/venv/bin"
ExecStart=/opt/aipm/app/backend/venv/bin/uvicorn main:app \
    --host 127.0.0.1 \
    --port 8000 \
    --workers 2
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

> `--workers 2`：2 个 worker 进程对 60 人的并发完全够用，内存占用约 200 MB。

```bash
# 启用并启动服务
sudo systemctl daemon-reload
sudo systemctl enable aipm-backend
sudo systemctl start aipm-backend

# 确认运行状态
sudo systemctl status aipm-backend
```

### 3.4 构建前端

```bash
cd /opt/aipm/app/frontend

# 安装依赖
npm install

# 构建生产版本
npm run build
# 构建产物输出到 frontend/dist/
```

### 3.5 配置 Nginx

```bash
sudo nano /etc/nginx/sites-available/aipm
```

填入：

```nginx
server {
    listen 80;
    server_name _;          # 替换为实际 IP 或内网域名，如 192.168.1.100

    # 前端静态文件
    root /opt/aipm/app/frontend/dist;
    index index.html;

    # Vue Router history 模式支持
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 反向代理到后端 API
    location /api/ {
        proxy_pass         http://127.0.0.1:8000;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_read_timeout 120s;
        client_max_body_size 20M;
    }
}
```

```bash
# 启用站点，禁用默认站点
sudo ln -s /etc/nginx/sites-available/aipm /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default

# 检查配置语法
sudo nginx -t

# 重载 Nginx
sudo systemctl reload nginx
sudo systemctl enable nginx
```

### 3.6 验证部署

在同网段另一台电脑浏览器访问 `http://<服务器IP>`，能看到登录页则部署成功。

默认管理员账号：
- 用户名：`admin`
- 密码：`admin123`

**首次登录后请立即修改默认密码。**

## 4. 数据备份与恢复

### 4.1 需要备份什么

| 文件 | 位置 | 重要性 |
|------|------|--------|
| `app.db` | `backend/app.db` | ★★★ 所有业务数据 |
| `config.json` | `backend/config.json` | ★★☆ 配置参数 |
| 报表文件 | `issue_report_path` 指定目录 | ★★☆ 原始数据 |

### 4.2 SQLite 在线备份

SQLite 提供 `.backup` 命令，支持**数据库运行时**安全备份（不需要停服）：

```bash
sqlite3 /opt/aipm/app/backend/app.db ".backup /opt/aipm/backups/app-$(date +%Y%m%d-%H%M%S).db"
```

### 4.3 自动定时备份（推荐）

创建备份脚本：

```bash
sudo mkdir -p /opt/aipm/backups
sudo chown $USER:$USER /opt/aipm/backups

nano /opt/aipm/backup.sh
```

脚本内容：

```bash
#!/bin/bash
set -e

BACKUP_DIR="/opt/aipm/backups"
DB_PATH="/opt/aipm/app/backend/app.db"
CFG_PATH="/opt/aipm/app/backend/config.json"
DATE=$(date +%Y%m%d-%H%M%S)

# 备份数据库（在线热备份）
sqlite3 "$DB_PATH" ".backup ${BACKUP_DIR}/app-${DATE}.db"

# 备份配置文件
cp "$CFG_PATH" "${BACKUP_DIR}/config-${DATE}.json"

# 保留最近 30 份，删除更旧的
find "$BACKUP_DIR" -name "app-*.db"       -mtime +30 -delete
find "$BACKUP_DIR" -name "config-*.json"  -mtime +30 -delete

echo "[$(date)] 备份完成：app-${DATE}.db"
```

```bash
chmod +x /opt/aipm/backup.sh

# 测试一次
/opt/aipm/backup.sh
```

注册 cron 定时任务（每天凌晨 2:00 执行）：

```bash
crontab -e
```

添加：

```
0 2 * * * /opt/aipm/backup.sh >> /var/log/aipm-backup.log 2>&1
```

### 4.4 从备份恢复

**停服 → 替换数据库文件 → 重启**：

```bash
# 1. 停止后端服务
sudo systemctl stop aipm-backend

# 2. 替换数据库（先保留损坏的文件以备查）
cp /opt/aipm/app/backend/app.db /opt/aipm/app.db.broken
cp /opt/aipm/backups/app-20260517-020001.db /opt/aipm/app/backend/app.db

# 3. 重启服务
sudo systemctl start aipm-backend
```

### 4.5 建议：异地备份

将 `/opt/aipm/backups/` 目录定期同步到另一台机器或 NAS，防止服务器硬盘故障导致数据和备份同时丢失：

```bash
# 使用 rsync 同步到另一台机器（每天凌晨 3:00）
# 在 crontab 中追加：
0 3 * * * rsync -az /opt/aipm/backups/ backup-server:/backup/aipm/
```

## 5. 升级流程

### 5.1 升级前准备

每次升级前务必先备份：

```bash
/opt/aipm/backup.sh
```

### 5.2 标准升级步骤

```bash
cd /opt/aipm/app

# 1. 拉取最新代码
git pull origin main

# 2. 更新后端依赖（若 requirements.txt 有变化）
cd backend
source venv/bin/activate
pip install -r requirements.txt

# 2.5 应用数据库迁移（v0.26.0 起引入 Alembic）
#     首次从 <v0.26 升上来的库，需先执行一次基线标记（只做一次）：
#       alembic stamp 0001_baseline
#     之后每次升级都执行：
alembic upgrade head

# 3. 重启后端（create_all 仍会自动建新表；数据迁移由上一步的 alembic 完成）
sudo systemctl restart aipm-backend
sudo systemctl status aipm-backend   # 确认启动正常

# 4. 重新构建前端
cd ../frontend
npm install           # 依赖有变化时自动处理
npm run build

# 5. 刷新 Nginx（静态文件已更新，无需重启，reload 即可）
sudo systemctl reload nginx
```

> 前端构建产物直接覆盖 `dist/`，Nginx reload 后用户刷新浏览器即可看到新版本。整个升级过程后端停服不超过 10 秒。

### 5.3 数据库结构变更

**v0.26.0 起改用 Alembic 管理结构变更。** 三层机制分工如下：

| 机制 | 负责 | 何时运行 |
|------|------|----------|
| `create_all`（`main.py` 启动时） | **新建表**（新增功能带来的整张表） | 重启后端自动 |
| Alembic（`backend/alembic/`） | **改名 / 删列 / 改类型 / 加约束 / 数据回填** | **手动** `alembic upgrade head` |
| `migrate.py`（已冻结） | 历史遗留的 `ADD COLUMN`，不再新增 | 重启后端自动 |

⚠️ **关键变化**：重启后端**不再自动完成全部迁移**——涉及数据改写的部分必须手动跑
`alembic upgrade head`（见 5.2 第 2.5 步）。新建表仍由 `create_all` 自动处理，所以
"只加了张新表"的版本即使忘了跑 alembic 也能用；但"改了已有数据"的版本必须跑。

首次从 v0.26 以前的库升级，需先做一次基线标记（**仅一次**）：

```bash
cd /opt/aipm/app/backend && source venv/bin/activate
alembic stamp 0001_baseline    # 告诉 Alembic：现有库就是基线状态
alembic upgrade head           # 再应用 0001 之后的迁移
alembic current                # 查看当前版本，确认 = head
```

> 升级前已用 `PRAGMA foreign_key_check` 核验生产库无悬空外键，启用外键强制是安全的。
> 若 `alembic upgrade` 报唯一约束/外键冲突，多半是存量脏数据，先按报错清理再重试。

若某次升级后出现 `500` 错误，优先查看后端日志：

```bash
sudo journalctl -u aipm-backend -n 100 --no-pager
```

### 5.4 特定版本升级说明

部分版本涉及数据格式或行为变更，升级时需额外注意。

---

#### v0.27.x → 专项分段可排序 + 附加表格升为独立分段 + 表格按列设格式

**变更内容**

1. **附加表格提升为独立分段**：专项/攻关详情页里，「专项事务」下方的附加表格（RichGrid）不再嵌在事务区块内部，
   而是**每张表各成一个与「专项事务」并列的独立分段**，标题即分段标题，右侧带「保存表格 / 删除整表」。
2. **表格支持按列设格式**：选中某列的正文单元格后，工具条「列格式」可把该列设为 **文本 / 下拉 / 日期**；
   选「下拉」时再填候选项（逗号分隔）。列格式存进 `extra_grids_json` 的 `colTypes` / `colOptions` 字段，
   随表持久化；插入 / 删除行列时自动对齐。旧表格无这两字段时按 `text` / `[]` 兜底。
3. **分段顺序可调（每个专项各自独立，编辑者可调）**：进入编辑模式后页面顶部出现「分段顺序」面板，
   可上下移动任意分段（含附加表格），例如把「求助」和「全景图」互换、把某张表格提到事务之前。
   顺序存进 `special_contents` 新增列 `section_order_json`（分段 key 的有序数组，空＝默认顺序）。
   附加表格分配了稳定 `gid` 作为分段 key（`grid:<gid>`），保存顺序时连同表格一起落库，跨刷新保持稳定。

**新增依赖**：无。

**新增表**：无。

**⚠️ 新增列（本版本必须手动跑 Alembic，`migrate.py` 已冻结不再自动加列）**
- `special_contents`：`section_order_json`（`TEXT NOT NULL DEFAULT '[]'`，迁移 `0003_special_section_order` 会给存量行回填 `'[]'`）

**升级步骤（⚠️ 这是继 v0.26 后又一个"改了已有表结构"的版本，务必跑迁移）**

```bash
cd /opt/aipm/app
/opt/aipm/backup.sh                       # 1. 先备份！
git pull origin main                      # 2. 拉代码

cd backend && source venv/bin/activate
alembic upgrade head                      # 3. ★必须★ 应用 0003（给 special_contents 加列）
alembic current                           # 确认版本 = head

sudo systemctl restart aipm-backend       # 4. 重启后端
cd ../frontend && npm install && npm run build   # 5. 重建前端
sudo systemctl reload nginx
```

**⚠️ 风险点：漏跑迁移会直接 500**

与"只加新表"的版本不同，本版给**已有表加了列**。`create_all` 不会给已有表补列，`migrate.py` 也已冻结不含此列，
所以**如果忘了 `alembic upgrade head` 就重启后端，专项详情 / 内容相关接口会因 SELECT 到不存在的
`section_order_json` 列而报 500**。升级后先用 admin 打开任意专项详情页确认能正常加载。

**验证**
- `alembic current` 显示已到 head；
- 打开一个专项详情页 → 进入编辑 → 顶部出现「分段顺序」面板，上下移动能改变分段显示顺序并保存；
- 「+ 添加表格」新增的表格作为独立分段出现在底部，可上移；选中某列可切「下拉 / 日期」格式。

**回滚**
- 回滚到 v0.26.x：还原 app.db 即可；或保留数据库、只回退代码——旧代码的 `SpecialContent` 模型不映射
  `section_order_json`，SQLAlchemy 只 SELECT 已映射列，**多出来的这一列对旧代码无害**。
  附加表格的 `colTypes` / `colOptions` 也会被旧版 RichGrid 忽略（按纯文本渲染，不丢数据）。
- 如需彻底清列：`alembic downgrade -1`（会 `DROP COLUMN section_order_json`）。

---

#### v0.26.x → 引入 Alembic + 启用外键约束 + 状态/优先级枚举统一

**变更内容**

1. **引入 Alembic** 作为数据库结构变更的系统（`backend/alembic/`，用法见 `backend/alembic/README.md`）。
   负责 `create_all` 做不到的改名 / 删列 / 改类型 / 加约束 / 数据回填；`migrate.py` 就此冻结，
   `create_all` 仍保留在启动路径建新表。**这是首个需要手动跑迁移的版本**（见下方升级步骤）。
2. **启用 SQLite 外键约束**：后端给每条连接加 `PRAGMA foreign_keys=ON`。此前 `ondelete=CASCADE/SET NULL`
   只是写进 DDL 不生效，现已在数据库层强制——删除被引用的主数据（如资源组）时，关联行的外键会按规则
   级联删除 / 置空，而不再留下指向已删行的"悬空外键"。
3. **状态/优先级枚举收口到 `backend/enums.py`**：进展状态 6 值（含「已变更」）、优先级 P0–P3 等统一管理。
   **两张需求表的优先级口径统一**：产品需求由「高/中/低」并入「P0–P3」，存量数据由迁移 `0002` 按
   高→P1 / 中→P2 / 低→P3 自动映射；Excel 导入遇旧值也自动转换，前端下拉同步只剩 P0–P3。

**新增依赖**：`alembic==1.13.2`。升级时 `pip install -r requirements.txt` 会自动装上。

**新增表**：无（不涉及新建业务表）。

**数据变更**：`iteration_product_requirements.priority` 历史「高/中/低」→「P1/P2/P3」（由 `0002` 迁移完成）。

**升级步骤（⚠️ 与以往不同，多了 Alembic 接入）**

```bash
cd /opt/aipm/app
/opt/aipm/backup.sh                       # 1. 先备份！
git pull origin main                      # 2. 拉代码

cd backend && source venv/bin/activate
pip install -r requirements.txt           # 3. 装 alembic

# 4. 一次性把现有库接入 Alembic（只在本次 v0.26 升级做一次）
alembic stamp 0001_baseline
alembic upgrade head                      # 应用 0002（优先级统一）
alembic current                           # 确认版本 = head

sudo systemctl restart aipm-backend       # 5. 重启（外键 pragma / 枚举校验生效）
cd ../frontend && npm install && npm run build
sudo systemctl reload nginx
```

> 之后的版本升级只需在标准流程里执行 `alembic upgrade head`（5.2 第 2.5 步），不必再 `stamp`。

**验证**

- `alembic current` 显示已到 head；
- 新建产品需求时优先级只剩 P0–P3；给需求传非法 priority/progress 返回 422；
- 删除一个被引用的资源组后，关联行的 `group_id` 在数据库里被置空（而非留悬空 id）。

**风险点**

- 这是**首个必须手动跑迁移**的版本，漏跑 `alembic upgrade head` 不会建表失败（无新表），
  但历史「高/中/低」的产品需求优先级不会被转换（生产库目前该表数据极少，影响有限）。
- 启用外键后，**删除主数据的行为更"严格"**：以往靠 ORM 兜底、裸 SQL 能删的悬空引用，现在会按

#### v0.10.x → 客户面状态清单字段格式变更

**变更内容**

`customer_status` 表的两个文本字段存储格式由纯文本改为 JSON 数组：

| 字段 | 旧格式（纯文本） | 新格式（JSON） |
|------|-----------------|----------------|
| `recent_focus` | `"良率提升\n备件采购"` | `[{"text":"良率提升","done":false},{"text":"备件采购","done":false}]` |
| `key_issues` | `"主控板偶发复位"` | `[{"text":"主控板偶发复位","done":false}]` |

**是否需要修改数据库？不需要。**

两个字段仍然是 `TEXT` 类型，表结构无任何变化。旧数据可直接使用，无需运行任何迁移脚本。

**兼容原理**

前端在读取字段时会自动判断格式：

```
读取字段值
    ├── 能解析为 JSON 数组 → 直接作为清单条目渲染（新格式）
    └── 否则按换行符拆分  → 每行作为一个未勾选条目渲染（旧格式自动兼容）
```

旧数据在用户第一次对该行执行操作（勾选、新增条目、点「保存」）后，才会被写成新 JSON 格式，之前一直保持原样。

**PPT 导出兼容**

`pptx_utils.py` 已同步更新，导出时自动将 JSON 清单转为可读文本：

```
✓ 已完成的条目
· 未完成的条目
```

旧纯文本字段原样输出，不受影响。

#### v0.24.x → 专项富表格列宽拖动 + 多人编辑锁

**变更内容**
- 「专项事务」下方的附加表格（RichGrid）支持**拖动表头右边缘调整列宽**；列宽随表持久化在 `extra_grids_json` 的 `colWidths` 字段，保存 / 刷新后保留，插入 / 删除行列时自动对齐。旧表格无 `colWidths` 时按默认列宽补齐。
- **多人并发编辑锁**：专项详情页同一时刻只允许一人处于编辑模式。
  - 进入编辑时向服务端取锁（`POST /api/specials/{id}/lock`），编辑期间每 60s 心跳续期，「完成编辑」或离开页面时释放。
  - 他人持锁时，本页顶部显示**只读横幅**（「XX 正在编辑…」），普通用户「进入编辑」按钮置灰；**管理员**可「强制接管」（接管后原编辑者再保存会被拒）。
  - 锁 **TTL 180s**：持锁人异常关闭页签（心跳停止）后，最长 3 分钟自动失效，他人即可接管，**不会出现永久死锁**。
  - 服务端在内容 / 全景图 / 事务 / 风险的所有写接口加了**前置校验**：他人持新鲜锁时返回 **423**，前端弹「正被他人编辑，暂时无法保存」。无人持锁时照常放行（向后兼容）。
- **修复内容串台**：`<router-view>` 改为按 `route.fullPath` 作 key，专项间切换（`/specials/1→/specials/2`）会重建组件实例，杜绝迟到异步响应把上一个专项的数据写到当前页；`load()` 另加请求令牌兜底。
- **Excel 导出优化**：
  - 「专项计划里程碑」由表格改为**时间轴图片**（PIL 绘制后嵌入，节点按状态着色 + 名称/日期 + 图例），不再是表格。
  - 「专项事务」下方的**附加表格现在会导出**：每个表 → 一张**独立工作表**（用各自 `colWidths` 定列宽，保留表头合并 / 对齐 / 字体颜色，**表头加粗**）。
- 附加表格在**页面上**的表头也强制加粗显示（原 600 → 700）。

**新增依赖**：`pillow>=9.0.0`（绘制里程碑图片；此前作为 python-pptx 的间接依赖已存在，现显式声明）。升级时 `pip install -r requirements.txt` 会自动装上。

**⚠️ Linux 服务器中文字体**：里程碑图片用 PIL 绘制中文，需系统装有中文字体，否则中文显示为方块。Ubuntu 安装：


**新增列**：无（`colWidths` 存在既有的 `extra_grids_json` 文本里，无需 schema 变更）。

**多 worker 注意**：编辑锁是**数据库表**实现，跨进程 / 跨 worker 一致，Uvicorn 开多 worker 也有效。

**变更内容**

1. **客户详情页结构调整**（路径不变，仍是 `/customers/:id`，全屏页面）：
   - 区块 1：基本信息（把"别名"合并进基本信息卡，不再独立成块）。
   - 区块 2：**客户面机台 tabs**（横向切换）。机台来自「客户面状态」总览中 `battlefield ∈ {customer.code, ...aliases}` 的所有行；每台机台一个 tab，tab 名 = `machine_id`。
   - 每个机台 tab 内含三个子区：
     - **当前进展**：编辑该机台的 `customer_status` 字段，与「客户面状态」总览**完全同步**（同一行、同一字段）。
     - **SOW 状态**：表格，**列由 admin 在「客户管理 → SOW 字段配置」全局维护**；所有客户共用同一套列定义。每一行的字段值按列定义动态渲染（文本/日期/下拉）。
     - **License 管理**：每台机台支持上传多个 license 文件 + 备注；普通用户可下载，admin 可上传/删除。
   - 区块 3：**问题单情况**（在 tabs 外，整客户范围）。从最新一期问题单报表中按客户编码/别名过滤展示，给出按严重程度的小计与明细。
2. 「客户面状态」页面的「返回」按钮指向**总览**而非「客户管理」（客户管理仅 admin 可见，客户详情所有用户都可看）。

**是否需要修改数据库？需要，但全部自动完成。**

- 新建三张表，由 SQLAlchemy `create_all` 在后端启动时创建：
  - `sow_field_defs`：全局 SOW 列定义（key / label / field_type / options / sort_order / is_active）。
  - `sow_rows`：每台机台下的 SOW 行；FK `machine_status_id → customer_status.id`，ON DELETE CASCADE；`data` 字段是 JSON Text 存 `{key: value}`。
  - `machine_licenses`：每台机台的 license 文件登记；FK `machine_status_id → customer_status.id`，ON DELETE CASCADE。
- **`customer_status` 表完全未动**；老数据照旧。
- 新上传目录：`backend/uploads/licenses/<machine_status_id>/<uuid>.<ext>`，首次上传时自动创建，权限沿用 `uploads/` 现有归属。

**对老数据的兼容**

- 新表不依赖任何老数据，可独立存在。
- 问题单视图复用现有 `/api/issues/data` 接口，按 `category`（标题分类）字段做 `customer.code` + 所有别名的大小写不敏感包含匹配；老数据中 `category` 与客户名一致的会被自动归并到对应客户。
- SOW 行通过 `machine_status_id` 与机台行强绑定，**如果在总览中删除该机台行，对应的 SOW 行和 license 文件登记会随级联删除一起消失**（license 物理文件随之孤立）；偶发误删可通过备份恢复，日常使用风险低。

**升级操作**

无额外步骤，按 5.2 标准流程执行：

```bash
git pull origin main
sudo systemctl restart aipm-backend     # 自动建 3 张表
cd frontend && npm install && npm run build
sudo systemctl reload nginx
```

**升级后建议**

1. admin 登录 → 「客户面状态 → 客户管理」→ 点击右上「SOW 字段配置」预先把 SOW 表头列建好（推荐：事项名称 / 计划完成 / 实际完成 / 状态 / 备注）。
2. 确认 `customer_status` 中各行的 `battlefield` 已经对应到正确的客户 code 或别名，否则机台 tabs 不会出现。
3. 确认 `issue_report_path` 已正确配置；问题单视图依赖最新一期报表的 `category` 字段。

**回滚说明**

如需回滚到 v0.16.x：

- 代码回退即可，数据库**无需回退**：旧代码不查 `sow_*` / `machine_licenses` 三张表，对正常业务无任何影响。
- 想彻底清理：停服后 SQLite 里执行
  ```sql
  DROP TABLE machine_licenses;
  DROP TABLE sow_rows;
  DROP TABLE sow_field_defs;
  ```
  以及删除 `backend/uploads/licenses/` 目录。

#### v0.11.x → 新增操作日志表 + 闲置自动登出

**变更内容**

1. 新增 `operation_logs` 表，记录登录与关键写操作（新增/修改/删除/导入/导出 PPT/运行脚本/修改配置 等），管理员可在前端「操作日志」页面分页查询、过滤。
2. 前端新增闲置自动登出：默认 **15 分钟** 无鼠标/键盘/滚动事件即清空会话并跳转登录页；多 tab 通过 `localStorage` 共享活动时间，任一 tab 在使用其它 tab 都不会被判定为闲置。
3. 任一 tab 主动登出（点「退出登录」、修改密码、收到 401）会通过 `localStorage` 事件广播给其它 tab 同步退出。

**是否需要修改数据库？不需要手动操作。**

`operation_logs` 表会在后端重启时由 SQLAlchemy 的 `create_all` 自动创建，与现有数据无冲突。

**注意事项**

- 操作日志只增不删，若长期运行体量较大，可在 SQLite 中按时间删除老数据：
  ```sql
  -- 例如保留最近 6 个月
  DELETE FROM operation_logs WHERE created_at < datetime('now','-6 months');
  VACUUM;
  ```
- 闲置阈值若需调整，修改 [frontend/src/App.vue](../frontend/src/App.vue) 中的常量 `IDLE_MS`。
- 前端会在「退出登录」时主动调用 `POST /api/auth/logout`，仅用于写一条登出日志，后端无服务端会话状态。

**升级操作**

无额外步骤，按 5.2 标准流程执行即可。

## 6. 常见问题排查

### 后端无法启动

```bash
sudo journalctl -u aipm-backend -n 50
```

常见原因：
- Python 版本不符：确认 `venv/bin/python --version` 为 3.11+
- 端口占用：`sudo ss -tlnp | grep 8000`
- `config.json` 格式错误：用 `python3 -m json.tool config.json` 验证

### 前端页面空白 / 404

```bash
sudo nginx -t                          # 检查 Nginx 配置语法
ls /opt/aipm/app/frontend/dist/        # 确认构建产物存在
sudo tail -f /var/log/nginx/error.log  # 查看 Nginx 错误日志
```

### API 请求 502 Bad Gateway

后端服务未运行或崩溃：

```bash
sudo systemctl status aipm-backend
sudo systemctl restart aipm-backend
```

### 忘记 admin 密码

直接在数据库中重置（需要 Python）：

```bash
cd /opt/aipm/app/backend
source venv/bin/activate
python3 - <<'EOF'
from auth import hash_password
from database import SessionLocal
import models

db = SessionLocal()
user = db.query(models.User).filter(models.User.username == "admin").first()
user.password_hash = hash_password("新密码")
db.commit()
print("密码已重置")
db.close()
EOF
```

### 查看实时日志

```bash
# 后端日志（实时）
sudo journalctl -u aipm-backend -f

# Nginx 访问日志
sudo tail -f /var/log/nginx/access.log
```
